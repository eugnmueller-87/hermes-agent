Database Layer

Two databases, two purposes — every crawled item lands in both simultaneously.

Upstash Redis
Exact lookup
hermes:item:{md5} ← full JSON, 7-day TTL
hermes:supplier:{slug} ← item ID list, max 3000
hermes:seen:{md5} ← dedup flag, 30-day TTL
"What does Hermes have on TSMC?" — slug lookup, returns last N items in milliseconds
Deduplication — URL hash stored before processing; never crawls the same article twice
Auto-expiry — 7-day TTL keeps storage lean; rolling window of fresh intelligence
Upstash Vector
Semantic search (RAG)
id: item MD5 hash
data: "{supplier} ({category}): {title}. {reason}"
metadata: supplier, category, urgency, date
model: BAAI/bge-large-en-v1.5 · 1024-dim
Text → embedding on write — Upstash hosts the model; no OpenAI key needed
Topic queries — "chip export controls" returns ASML + NVIDIA + Applied Materials without naming them
10,000 vectors free — cosine similarity, EU region, zero config
Crawler → Signal Detection (Claude Haiku) → Redis write (key-value) → Vector upsert (embed text) → Both databases updated in one store_item() call

RAG — Semantic Search Across All Suppliers

Retrieval-Augmented Generation: retrieve relevant items by meaning, not by company name.

Before RAG — keyword lookup
"What does Hermes have on NVIDIA?"
Returns NVIDIA-specific items only.

Fails for: "what's happening with chip supply?", "export control signals", "AI compute capacity" — no company name, no results.
After RAG — semantic retrieval
"What are chip suppliers signaling about export controls?"
Returns relevant items from ASML, NVIDIA, Applied Materials, KLA, TSMC — based on semantic similarity, not name matching.
